## 1. 系统架构总览（纯Python双核隔离高性能版）

### 1.1 架构拓扑图（双核隔离主从进程架构）

**核心设计理念**：将I/O密集型任务（WebSocket、状态机、风控）与CPU密集型任务（KDE计算）严格隔离到两个不同的CPU核心，确保单Tick处理延迟<1ms。

```text
┌─────────────────────────────────────────────────────────────────────────────┐
│                           OKX WebSocket Gateway                             │
└──────────────────────────────────┬──────────────────────────────────────────┘
                                   │ Tick Stream (JSON)
                                   ▼
┌─────────────────────────────────────────────────────────────────────────────┐
│                            主进程 (核心 1: I/O密集)                          │
│  ┌───────────────────────────────────────────────────────────────────────┐  │
│  │ asyncio 事件循环                                                      │  │
│  │ ├─ WebSocket Handler (aiohttp)                                        │  │
│  │ ├─ Range Bar Generator (物理位移计算)                                   │  │
│  │ ├─ Triple-A State Machine (毫秒级状态切换)                              │  │
│  │ └─ Risk & Execution Manager (风控与下单)                               │  │
│  └──────────────────────────────────┬────────────────────────────────────┘  │
│                                     │ IPC (进程间通信: 切片数据 / LVN结果)      │
│                                     ▼                                       │
│  ┌───────────────────────────────────────────────────────────────────────┐  │

                                   │ TradeCommand
                                   ▼
┌─────────────────────────────────────────────────────────────────────────────┐
│                      OKX REST API Executor (异步执行)                      │
│                    /api/v5/trade/order (市价单/IOC)                        │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 1.2 核心设计原则

1. **CPU与IO严格隔离**：I/O操作（WebSocket、状态机）在主进程asyncio事件循环中运行，CPU密集型计算（KDE）在独立的进程池中执行，避免阻塞主循环。

2. **矩阵化与预编译**：废弃所有Python原生的`for`循环遍历Tick进行数学计算，使用Numpy广播机制，并对高频调用的数学函数使用`@njit(cache=True)`进行AOT/JIT编译。

3. **Python原生性能优化**：使用`numpy`进行批量计算，`dataclasses`定义数据结构，`asyncio`处理异步，`ProcessPoolExecutor`进行多进程隔离。

4. **内存高效管理**：使用预分配的`deque`和`list`缓冲区，避免频繁内存分配，定期清理历史数据。

5. **事件驱动状态机**：状态转换由Tick事件触发，**严禁**使用定时器驱动，确保响应速度。

6. **单向数据流**：策略层只发命令，不等待执行结果，执行状态由独立监控协程跟踪。

7. **小资金优化**：专为300U-10万U设计的风险参数（5%单笔风险，2 tick止损，6 tick止盈）。

## 3. 核心算法实现（纯Python高性能优化版）

### 3.1 Range Bar生成器（Algorithm 1 - Python实现）

**输入**: `NormalizedTick`流
**输出**: `RangeBar`流
**触发条件**: 价格物理位移达到`tick_range` (20 Ticks)

```python
class RangeBarGenerator:
    """Range Bar生成器（Python实现）"""

    def __init__(self, config: RangeBarConfig):
        self.config = config
        self.current_bar: Optional[RangeBar] = None
        self.open_px_base: float = 0.0
        self.bar_history: deque = deque(maxlen=config.max_bar_history)

    def on_tick(self, tick: NormalizedTick) -> Optional[RangeBar]:
        """处理单个Tick，可能返回闭合的Bar"""
        if self.current_bar is None:
            # 初始化新Bar
            self.current_bar = RangeBar(
                open_ts=tick.ts,
                open_px=tick.px,
                high_px=tick.px,
                low_px=tick.px,
                close_px=tick.px,
                total_buy_vol=tick.sz if tick.side == 1 else 0.0,
                total_sell_vol=tick.sz if tick.side == -1 else 0.0,
                delta=tick.sz if tick.side == 1 else -tick.sz,
                tick_count=1,
                is_closed=False
            )
            self.open_px_base = tick.px
            return None

        # 更新当前Bar
        self.current_bar.high_px = max(self.current_bar.high_px, tick.px)
        self.current_bar.low_px = min(self.current_bar.low_px, tick.px)
        self.current_bar.close_px = tick.px

        if tick.side == 1:
            self.current_bar.total_buy_vol += tick.sz
        else:
            self.current_bar.total_sell_vol += tick.sz

        self.current_bar.delta = self.current_bar.total_buy_vol - self.current_bar.total_sell_vol
        self.current_bar.tick_count += 1

        # 检查是否触发闭合条件
        displacement = abs(tick.px - self.open_px_base) / self.config.tick_range
        if displacement >= 1.0:  # 达到20个Tick位移
            return self._close_bar_and_emit(tick)

        return None

    def _close_bar_and_emit(self, overflow_tick: NormalizedTick) -> RangeBar:
        """闭合当前Bar，处理溢出Tick"""
        if self.current_bar is None:
            return None

        self.current_bar.is_closed = True
        completed_bar = self.current_bar
        self.bar_history.append(completed_bar)

        # 新开Bar，处理溢出
        self.current_bar = RangeBar(
            open_ts=overflow_tick.ts,
            open_px=overflow_tick.px,
            high_px=overflow_tick.px,
            low_px=overflow_tick.px,
            close_px=overflow_tick.px,
            total_buy_vol=overflow_tick.sz if overflow_tick.side == 1 else 0.0,
            total_sell_vol=overflow_tick.sz if overflow_tick.side == -1 else 0.0,
            delta=overflow_tick.sz if overflow_tick.side == 1 else -overflow_tick.sz,
            tick_count=1,
            is_closed=False
        )
        self.open_px_base = overflow_tick.px

        return completed_bar
```

### 3.2 KDE与LVN提取引擎（Algorithm 2 - Python高性能优化版，Numba JIT编译）

**输入**: `RangeBar`序列（脉冲波段）
**输出**: `LVN_Zone`数组（最多3个区域）
**性能优化**: 使用Numba JIT编译 + Numpy矩阵广播，消除Python级循环

```python
import numpy as np
from numba import njit
from concurrent.futures import ProcessPoolExecutor
import asyncio
from typing import List, Tuple

# ==================== 核心计算函数（Numba JIT编译）====================
# 必须放在类外部作为独立纯函数，以便 Numba 编译和跨进程传递
@njit(cache=True)
def _compute_kde_core(prices: np.ndarray, volumes: np.ndarray, price_range: np.ndarray, h: float) -> np.ndarray:
    """
    使用 Numba 和 Numpy 广播加速的核心 KDE 计算（无Python级循环）

    算法原理：
    1. 巧妙利用矩阵广播：prices扩展为行向量，price_range扩展为列向量
    2. 计算出的 diff 矩阵形状为 (len(price_range), len(prices))
    3. 一次性算出所有点对的高斯核
    4. 利用矩阵点乘直接完成体积加权求和，速度提升百倍

    性能对比：
    - 原始Python循环: O(n*m) 纯Python循环，极慢
    - Numpy广播 + Numba: O(n*m) 但为机器码级执行，速度提升100-1000倍
    """
    # 矩阵广播计算所有价格对的距离
    # price_range形状: (R, 1), prices形状: (1, N) -> diff形状: (R, N)
    diff = np.expand_dims(price_range, 1) - prices

    # 一次性算出所有点对的高斯核（向量化计算）
    kernels = np.exp(-0.5 * (diff / h) ** 2) / (h * np.sqrt(2 * np.pi))

    # 利用矩阵点乘直接完成体积加权求和
    density = np.dot(kernels, volumes)
    return density

@njit(cache=True)
def _find_local_minima_fast(density: np.ndarray) -> np.ndarray:
    """使用Numba加速的局部极小值查找"""
    minima = []
    for i in range(1, len(density) - 1):
        if density[i] < density[i-1] and density[i] < density[i+1]:
            minima.append(i)
    return np.array(minima)

class KDEEngine:
    """KDE引擎（高性能优化版，支持进程池异步计算）"""

    def __init__(self, config: KDEEngineConfig):
        self.config = config

    # 这里的函数设计为静态方法，方便放进进程池（无self状态依赖）
    @staticmethod
    def extract_lvn_heavy_lifting(prices: np.ndarray, volumes: np.ndarray, tick_size: float, config: KDEEngineConfig) -> List[Tuple[float, float]]:
        """
        KDE核心计算函数（在独立的CPU核心/子进程中运行）

        注意：此函数将在ProcessPoolExecutor中执行，必须：
        1. 不依赖self状态（静态方法）
        2. 参数和返回值可pickle序列化
        3. 避免使用全局变量
        """
        if len(prices) < config.min_slice_ticks:
            return []

        # Step 1: 稳健Silverman带宽计算（保持原有逻辑）

        std_dev = np.std(prices)
        iqr = np.percentile(prices, 75) - np.percentile(prices, 25)
        n = len(prices)

        # 防止除以零
        if iqr == 0:
            iqr = std_dev * 0.5  # 备用方案

        # Silverman法则
        h = 0.9 * min(std_dev, iqr / 1.34) * (n ** -0.2)

        # 限制带宽范围（防止极端值）
        h = max(h, tick_size * 2)  # 最小2个Tick
        h = min(h, (prices.max() - prices.min()) * 0.1)  # 最大10%范围

        # Step 2: 构建价格评估网格
        price_range = np.linspace(
            prices.min() - 5 * tick_size,
            prices.max() + 5 * tick_size,
            int((prices.max() - prices.min()) / tick_size) + 10
        )

        # Step 3: 调用被 Numba 编译成机器码的核心函数（性能关键）
        density = _compute_kde_core(prices, volumes, price_range, h)

        # Step 4: 寻找局部极小值（Numba加速）
        minima_indices = _find_local_minima_fast(density)

        # Step 5: 密度阈值过滤（<30%分位）
        density_threshold = np.percentile(density, config.lvn_density_percentile)
        lvn_zones = []

        for idx in minima_indices:
            if density[idx] < density_threshold:
                center_price = price_range[idx]
                lvn_zones.append((
                    center_price - tick_size,
                    center_price + tick_size
                ))

        # 最多返回3个，按密度升序排序
        lvn_zones = sorted(lvn_zones, key=lambda x: x[0])[:config.max_lvn_zones]

        return lvn_zones

    def extract_lvn(self, slice_data: ImpulseSlice, tick_size: float = 0.01) -> List[Tuple[float, float]]:
        """
        主入口函数（兼容原有接口）
        注意：实际实盘中将通过进程池调用 extract_lvn_heavy_lifting
        """
        # 提取数据
        prices = np.array([tick.px for tick in slice_data.ticks])
        volumes = np.array([tick.sz for tick in slice_data.ticks])

        # 调用静态方法（保持接口一致）
        return self.extract_lvn_heavy_lifting(prices, volumes, tick_size, self.config)
```

### 3.3 Triple-A状态机（Algorithm 3 - Python实现）

**状态定义**:

```python
class TripleAState:
    """Triple-A状态枚举"""
    IDLE = "IDLE"                       # 休眠：价格不在任何LVN区域内
    MONITORING_ABSORPTION = "MONITORING_ABSORPTION"      # 监测中：价格进入LVN，监测努力vs结果
    ABSORPTION_CONFIRMED = "ABSORPTION_CONFIRMED"       # 吸收确认：检测到高CVD低位移
```


**详细判定逻辑**:
```python
class TripleAStateMachine:
    """Triple-A状态机（Python实现）"""

    def __init__(self, config: TripleAConfig):
        self.config = config
        self.ctx = TripleAContext()
        self.kde_engine = KDEEngine(KDEEngineConfig())
        self.range_bar_gen = RangeBarGenerator(RangeBarConfig())
        self.historical_cvd = deque(maxlen=1000)  # 历史CVD数据

    async def on_normalized_tick(self, tick: NormalizedTick):
        """处理标准化Tick（异步）"""
        self._update_context(tick)

        if self.ctx.state == TripleAState.IDLE:
            if self._is_price_in_lvn(tick.px):
                await self._transition_to(TripleAState.MONITORING_ABSORPTION, tick)

        elif self.ctx.state == TripleAState.MONITORING_ABSORPTION:
            # 检查双维度超时
            time_elapsed_ms = (tick.ts - self.ctx.entry_ts) / 1_000_000
            tick_timeout = self.ctx.tick_count_in_lvn >= self.config.absorption_window_ticks
            time_timeout = time_elapsed_ms > 30000  # 30秒强制超时

            if tick_timeout or time_timeout:
                if not (effort_ok and result_ok and efficiency_ok):
                    await self._transition_to(TripleAState.IDLE, tick)
                    return
            # 累积努力与结果数据
            self.ctx.effort_cvd += tick.side * tick.sz
            self.ctx.result_delta_px = tick.px - self.ctx.entry_px
            self.ctx.tick_count_in_lvn += 1

            # 检查是否满足吸收条件
            if self.ctx.tick_count_in_lvn >= self.config.absorption_window_ticks:
                cvd_mean = self._get_historical_cvd_mean(self.config.absorption_window_ticks)
                cvd_std = self._get_historical_cvd_std(self.config.absorption_window_ticks)
                zscore = (self.ctx.effort_cvd - cvd_mean) / cvd_std if cvd_std > 0 else 0

                effort_ok = abs(zscore) > self.config.effort_zscore_threshold
                result_ok = abs(self.ctx.result_delta_px) <= self.config.max_result_px_deviation
                efficiency_ok = abs(self.ctx.result_delta_px / max(self.ctx.effort_cvd, 1e-8)) > self.config.cvd_efficiency_min

                if effort_ok and result_ok and efficiency_ok:
                    await self._transition_to(TripleAState.ABSORPTION_CONFIRMED, tick)
                elif not self._is_price_in_lvn(tick.px):
                    await self._transition_to(TripleAState.IDLE, tick)

        elif self.ctx.state == TripleAState.ABSORPTION_CONFIRMED:
            # 监测积累条件
            self.ctx.acc_tick_count += 1
            self._update_accumulation_volatility(tick.px)

            range_ok = self._get_current_range_ticks() <= self.config.accumulation_max_range_ticks
            density_ok = self.ctx.acc_tick_count >= self.config.accumulation_min_tick_count
            timeout = (tick.ts - self.ctx.entry_ts) > self.config.accumulation_timeout_ms * 1_000_000

            if range_ok and density_ok:
                await self._transition_to(TripleAState.ACCUMULATING, tick)
            elif self._get_current_range_ticks() > self.config.accumulation_max_range_ticks or timeout:
                await self._transition_to(TripleAState.IDLE, tick)

        elif self.ctx.state == TripleAState.ACCUMULATING:
            # 监测攻击信号
            if self._detect_aggression(tick):
                self.ctx.trigger_px = tick.px
                self.ctx.aggression_sz = self._calculate_bubble_size(tick)
                await self._transition_to(TripleAState.AGGRESSION_TRIGGERED, tick)
            elif not self._is_price_in_lvn(tick.px):
                await self._transition_to(TripleAState.IDLE, tick)

        elif self.ctx.state == TripleAState.AGGRESSION_TRIGGERED:
            # 已发送交易指令，等待执行层确认
            # 此状态下不再处理新Tick，由RiskManager接管
            pass

    def _detect_aggression(self, tick: NormalizedTick) -> bool:
        """攻击信号检测"""
        # 1. 大单气泡检测（动态阈值）
        bubble_threshold = self._get_percentile_volume(self.config.aggression_print_percentile)
        bubble_ok = tick.sz > bubble_threshold

        # 2. 足迹失衡检测（3档连续300%失衡）
        imbalance_ok = self._check_footprint_imbalance(
            tick,
            self.config.aggression_imbalance_ratio,
            self.config.aggression_stacked_levels
        )

        # 3. CVD斜率确认
        current_slope = self._calculate_cvd_slope(tick)
        slope_ok = abs(current_slope) > self.config.aggression_cvd_slope_threshold

        # 方向一致性检查（与宏观状态一致）
        direction_ok = (tick.side == self._get_macro_direction())

        return (bubble_ok or imbalance_ok) and slope_ok and direction_ok

    async def _transition_to(self, new_state: str, tick: NormalizedTick):
        """状态转换"""
        old_state = self.ctx.state
        self.ctx.state = new_state
        self.ctx.entry_ts = tick.ts
        self.ctx.entry_px = tick.px

        # 状态特定的初始化
        if new_state == TripleAState.MONITORING_ABSORPTION:
            self.ctx.effort_cvd = 0.0
            self.ctx.result_delta_px = 0.0
            self.ctx.tick_count_in_lvn = 0

        elif new_state == TripleAState.ABSORPTION_CONFIRMED:
            self.ctx.acc_tick_count = 0
            self.ctx.acc_volatility = 0.0

        elif new_state == TripleAState.AGGRESSION_TRIGGERED:
            # 触发交易执行
            await self._execute_trade_signal(tick)
```

```python
        self.daily_pnl = 0.0
        self.daily_trades = 0

   def calculate_position(self, entry_px: float, structure_sl_px: float, structure_tp_px: float, equity: float, direction: int) -> PositionSizingResult:
        """
        基于结构位和真实手续费的动态仓位计算
        """
        # 1. 计算物理距离
        sl_distance = abs(entry_px - structure_sl_px)
        tp_distance = abs(structure_tp_px - entry_px)
        
        # 2. 计算 5% 的绝对风险金额 (例: 300U * 5% = 15U)
        risk_amount = equity * self.config.risk_per_trade_pct
        
        # 3. 计算包含手续费的真实合约数量
        # 公式: Risk = Qty * (SL_Distance + Entry_Px * Taker_Fee + SL_Px * Taker_Fee)
        # 估算简写: Qty = risk_amount / (sl_distance + entry_px * fee_rate * 2)
        estimated_fee_per_coin = entry_px * self.config.fee_rate_taker * 2
        qty = round(risk_amount / (sl_distance + estimated_fee_per_coin), 3)
        
        # 4. 强制手续费与盈亏比拦截 (核心护城河)
        expected_gross_profit = qty * tp_distance
        total_estimated_fee = qty * estimated_fee_per_coin
        expected_net_profit = expected_gross_profit - total_estimated_fee
        
        # 拦截逻辑：如果连赚头都不够塞牙缝，直接拒绝开仓
        if expected_net_profit <= 0:
            logger.warning("风控拦截：预期利润无法覆盖双边手续费！")
            return PositionSizingResult(0.0, 0.0, 0.0, 0.0)
            
        if (expected_net_profit / risk_amount) < self.config.min_rr_ratio:
            logger.warning(f"风控拦截：净盈亏比不足 {self.config.min_rr_ratio}！")
            return PositionSizingResult(0.0, 0.0, 0.0, 0.0)

        # 5. 返回计算结果 (保本价设为覆盖单边手续费+1个Tick)
        breakeven_offset = (entry_px * self.config.fee_rate_taker) + 0.01
        breakeven_px = entry_px + breakeven_offset if direction == 1 else entry_px - breakeven_offset
        
        return PositionSizingResult(qty, structure_sl_px, structure_tp_px, breakeven_px)
    def update_daily_pnl(self, pnl: float):
        """更新日盈亏"""
        self.daily_pnl += pnl
        self.daily_trades += 1
```

```python
        self.side = side
        self.qty = qty

        self.highest_cvd_since_entry = await cvd_getter()
        self.highest_px_since_entry = entry_px

    async def on_tick(self, tick: NormalizedTick):

        if self.side == 1 and tick.px > self.entry_px:  # 多单浮盈中
            current_cvd = await self.current_cvd_getter()
            self.highest_cvd_since_entry = max(self.highest_cvd_since_entry, current_cvd)
            self.highest_px_since_entry = max(self.highest_px_since_entry, tick.px)

            # 检测CVD背离：价格新高但CVD未新高
            price_new_high = tick.px > self.highest_px_since_entry * 0.998  # 0.2%容差
            cvd_not_new_high = current_cvd < self.highest_cvd_since_entry * 0.995  # 0.5%容差

            if price_new_high and cvd_not_new_high:
                # 触发保本：将止损移至保本价
                return True  # 需要调整止损

        return False  # 不需要调整
```

## 8. 注意事项（纯Python高性能优化特别指导）

### 8.1 性能关键点（v3.0增强版）

1. **CPU与IO严格隔离**：严禁在 `asyncio` 的主事件循环中执行耗时超过 2ms 的数学运算。KDE 计算必须通过 `ProcessPoolExecutor` 卸载到独立进程，绑定到专用CPU核心。

2. **矩阵化与预编译**：废弃所有 Python 原生的 `for` 循环遍历 Tick 进行数学计算。必须使用 Numpy 广播机制，并对高频调用的数学函数使用 `@njit(cache=True)` 进行 AOT/JIT 编译。

3. **避免热路径GC**：使用预分配的 `deque` 和 `list` 缓冲区，避免频繁创建新对象。对于高频数据结构，考虑使用 `array.array` 或 `numpy.ndarray` 预分配内存。

4. **数值计算优化**：使用 `numpy` 进行批量计算，避免Python循环。对于标量计算，使用 `math` 模块而非 `numpy`。

5. **异步非阻塞**：所有I/O操作使用 `asyncio`，避免阻塞事件循环。网络请求、磁盘写入等必须异步执行。

6. **内存管理**：定期清理历史数据，防止内存泄漏。使用 `weakref` 管理循环引用，监控进程池内存使用。

### 8.5 Numba冷启动问题与热身策略

在基于此文档进行代码实现时，有一个Numba JIT编译的**实战避坑指南**需要注意：**JIT编译冷启动耗时问题**。

#### 问题描述
`@njit`装饰的函数在第一次被调用时，Numba会实时将其翻译成LLVM IR并编译成机器码，这个过程大约需要**200ms - 500ms**。如果系统刚启动，遇到的第一个真实脉冲波就去触发编译，会导致那一次的KDE计算极度卡顿，可能错过关键交易机会。

#### 解决方案：预编译热身（Warm-up）
在系统启动初始化的阶段（`TripleAEngine`的`__init__`之后，或进入WebSocket接收循环之前），手动构造一组假数据（Dummy Data）去调用一次KDE计算函数，完成"热身"。

```python
# 伪代码：引擎启动时的热身 (Warm-up)
def _warmup_numba_compiler(self):
    logging.info("正在预编译 KDE 核心引擎...")
    dummy_prices = np.linspace(1000, 1010, 100)
    dummy_volumes = np.random.random(100)
    # 丢进进程池空跑一次
    self.process_pool.submit(
        KDEEngine.extract_lvn_heavy_lifting,
        dummy_prices, dummy_volumes, 0.01, self.config.kde_engine
    )
    logging.info("KDE 引擎预编译完成，随时准备战斗。")
```

#### 实现位置建议
1. **在`TripleAEngine.__init__`末尾调用**：确保引擎初始化完成立即热身
2. **异步执行**：使用`asyncio.create_task`异步执行热身，不阻塞主线程
3. **超时保护**：设置热身超时（如2秒），防止意外卡死
4. **失败降级**：热身失败时记录警告，但不影响系统启动

#### 性能收益
- **首次KDE计算延迟**：从200-500ms降至<5ms
- **系统稳定性**：避免突发性计算卡顿
- **实时性保障**：确保第一个真实脉冲波就能得到毫秒级响应
